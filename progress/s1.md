# Task 1 — Multi-Tenant Backend Foundation (Infrastructure + Database + Security)

## Overview
Task 1 focused on building the foundational backend architecture for a multi-tenant AI system.  
The goal was to design a system where multiple customers (tenants) share the same infrastructure while remaining fully isolated at the database level.

This phase established:
- Docker-based infrastructure (Postgres + Redis)
- Database schema design using Alembic migrations
- Multi-tenant architecture using PostgreSQL Row Level Security (RLS)
- AI-ready database foundation (pgvector support)

---

## 1. Infrastructure Setup (Docker)

### What was built
- PostgreSQL (pgvector enabled)
- Redis (for future async queue processing)
- Persistent volumes for data safety

### Why this approach
- **Docker** ensures environment consistency across dev/prod
- **PostgreSQL** chosen for reliability and strong relational guarantees
- **pgvector extension** enables semantic search for AI features later
- **Redis** prepares system for async job pipelines (classification, embedding, etc.)

### Key learning
Initial issues included missing docker-compose configuration and container naming confusion. These were resolved by standardizing service definitions and ensuring correct container orchestration.

## 2. Database Migration System (Alembic)

### What was built
- Alembic initialized for schema versioning
- First migration (`0001_initial`) created

### Why Alembic

## 3. Core Schema Design

### Tables created
- tenants
- tickets
- kb_articles
- replies
- feedback_events

### Design decisions

#### UUID Primary Keys
All tables use UUID instead of BIGINT.

**Why UUID:**
- Globally unique (important for distributed systems)
- Avoids ID collision across services
- Better suited for multi-tenant SaaS architecture
- Works well with RLS policies

---

#### Core Data Model Philosophy
- Every table is tenant-scoped
- Every row contains `tenant_id`
- No cross-tenant shared state

---

## 4. Row Level Security (RLS) — Core Security Layer

### What was implemented
PostgreSQL RLS enabled on all tenant-scoped tables:

```sql
tenant_id = current_setting('app.tenant_id')::uuid
```

### How it works
- Each request sets a session variable:
  `SET app.tenant_id = '<tenant_uuid>'`
- PostgreSQL automatically filters all queries

### Why RLS instead of app-level filtering
- Enforces security at database level (not optional)
- Prevents accidental cross-tenant leaks
- Works even if application layer has bugs
- Industry-grade SaaS pattern (used in Stripe-like systems)

---

## 5. Security Hardening

### Improvements made
- Enabled BOTH:
  - `USING` (read/update/delete filtering)
  - `WITH CHECK` (insert validation)

### Why both are required
- `USING` → controls visibility
- `WITH CHECK` → prevents inserting invalid tenant data

This ensures complete tenant isolation.

---

## 6. Key Challenges & Fixes

### Issue 1: UUID vs BIGINT mismatch
**Problem:** Mixed ID types caused insertion failures  
**Fix:** Standardized everything to UUID

---

### Issue 2: Missing extensions
**Problem:** pgcrypto and vector not enabled  
**Fix:** Enabled:
- pgcrypto → UUID generation
- vector → embeddings support

---

### Issue 3: RLS confusion during testing
**Problem:** Superuser bypassed RLS  
**Fix:** Switched to proper DB role (`app_user`) instead of superuser

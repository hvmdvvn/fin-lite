
# RAG Core Pipeline (Multi-Tenant AI Ticket System)

## Overview

Focuses on building the **core Retrieval-Augmented Generation (RAG) pipeline** that powers intelligent ticket responses.

The system evolves from simple ticket storage into a **multi-stage AI processing pipeline**:

```

Ticket → Classification → Retrieval (KB Search) → Draft Generation (LLM)

```

All components are **multi-tenant safe using PostgreSQL Row Level Security (RLS)** and coordinated through **Redis-based job queues**.

---

# Architecture Summary

```

FastAPI API Layer

▼

▼

│
▼
Worker Agents
│   ├── Classifier

# Completed Components

## 1. KB Embedding Pipeline

### Purpose
Convert knowledge base articles into embeddings for semantic search.

### Implementation
- SentenceTransformer model used (`all-MiniLM-L6-v2`)
- Embeddings stored in PostgreSQL `vector(384)`
- Fixed dimension mismatch (1536 → 384)

### File:
```

app/rag/embed.py

````

### Key Features:
- Local embedding model (no external API dependency)
- Fast inference on CPU/GPU
- Compatible with pgvector cosine similarity

---

## 2. KB Article API

### Endpoint
```

POST /kb/articles

````

### Behavior:

* Accepts `{title, body}`
* Automatically:

  * chunks (basic MVP approach)
  * generates embeddings
  * stores in DB with tenant isolation

### Table:

* `kb_articles`
* `embedding vector(384)`
* RLS enabled

---

## 3. Ticket Ingestion API

### Endpoint

```

POST /tickets

```

### Flow:

1. Accepts `{subject, body}`
2. Extracts tenant from JWT via RLS context
3. Inserts ticket into database
4. Pushes job into Redis `classify` queue
5. Returns `202 Accepted`

### Example response:

```json
{
  "ticket_id": "uuid",
  "status": "new"
}
```

---

## 4. Redis Queue System

### Implementation

Simple Redis-based queue system using:

* `LPUSH` → enqueue
* `BRPOP` → worker consume

### Queues:

```

classify → retrieve → draft

```

### File:

```

app/core/queue.py

```

### Functions:

* `enqueue_classify(ticket_id)`
* `get_next_classify_job()`

---

## 5. Classifier Worker

### Purpose

Assigns a category to incoming tickets.

### Flow:

1. Reads from `classify` queue
2. Processes ticket
3. Assigns category (MVP: static "general")
4. Enqueues next step → `retrieve`

---

## 6. Retriever (RAG Core)

### Purpose

Fetch relevant KB articles using semantic similarity.

### Technology:

* pgvector cosine similarity
* tenant-scoped queries via RLS

### Query:

```sql
SELECT *
FROM kb_articles
ORDER BY embedding <=> :query_embedding
LIMIT 5;
```

### Output:

* Ranked KB chunks
* Passed to drafter stage

---

## 7. Drafter Agent (LLM Layer)

### Purpose

Generate final response using retrieved context.

### Flow:

1. Receives chunk IDs
2. Fetches KB content
3. Calls LLM (Groq / OpenAI / etc.)
4. Produces structured response:

```json
{
  "draft": "...",
  "used_chunk_ids": [...]
}
```

5. Stores result in `replies` table

---

## 8. Row Level Security (RLS)

### Applied to:

* `tickets`
* `kb_articles`
* `replies`
* `feedback_events`

### Policy:

```sql
USING (tenant_id = current_setting('app.tenant_id')::uuid)
WITH CHECK (tenant_id = current_setting('app.tenant_id')::uuid)
```

### Result:

* Hard tenant isolation at DB level
* No cross-tenant leakage possible

---

## 9. Tenant Context Injection

### Implementation:

FastAPI dependency:

```python
SET LOCAL app.tenant_id = :tenant_id
```

Ensures:

* Every DB query is tenant-scoped automatically
* RLS enforced per request
